# 脱离 Python 产品路线图

> 视角：产品经理。一切排序服务于**唯一的北极星**，不服务于「让某个测试变绿」。
> 时间：2026-06-15　分支：`bootstrap-ku-bytecode`　基线 commit：`b5d63f2`（M0/M1/M2 已干净绿）

---

## 0. 北极星与价值主张

**用户**：玄璃 / 天书作者（唯一用户）。

**最高产品定义**：Dao/Ku 不是普通编程语言，也不是单纯解释器工程；它是给 AI 准备的**可执行大脑记忆库**。代码工程本身就是 AI 的记忆结构：代码、数据集、经验、失败、任务、推理轨迹、工具调用都应成为可加载、可执行、可演化的记忆单元。未来目标是让 AI 在任何地点、面对任何领域，都能迅速加载相关记忆、理解环境、调用能力、积累经验，逐步向强人工智能 / 通用人工智能靠近。

**层级关系**：

- **终点**：AI 大脑记忆库独立、可携带、可执行、可积累、可自举。
- **手段**：玄璃执行路径脱离 Python，C VM 接管，经验层和 MCP 接口落到原生内核上。
- **反例**：为了测试绿而保留 Python fallback，只是在产品形态里固化外部拐杖，不等于独立站立。

**真正的需求**（用户原话）：

> 「目标是**脱离 Python**，为什么还在用 Python？」

**价值主张**：天书母语要**独立站立**，不寄生在 CPython 解释器上。
这是一个**主权 / 自举**诉求，不是性能诉求。性能、内存只是「独立站立」的副产品要求，不是目的。

## 2026-07-05 reality calibration

This roadmap predates the current M3/M5 implementation state. Current code reality:

- `dao_core.exe` can run `golden_path.ku`, `semantic_std_combo.kub.json`, and `frontend_compile_demo.kub.json` directly on Windows.
- The C VM includes SQLite, `now` / `now_fmt`, and `dao_data_path`, enough for the main `experience.ku` and `task_queue.ku` paths.
- The MCP gateway now calls the C VM through `CVMRuntime` by default. Python is shrinking toward stdio glue, schema discovery, parity tests, and temporary fallback scaffolding.
- `ku_eval` no longer silently falls back to Python when the C VM binary is missing; fallback now requires `DAO_MCP_ALLOW_PYTHON_FALLBACK=1`.
- The experience layer has been manually verified, and now has a regression test, for UTF-8 Chinese topics persisted through SQLite.
- `tools/test.ps1` is the fixed local Windows test entry: build C VM, validate the TextMate grammar JSON, then run pytest.

So the next work should no longer be ordered as "implement SQLite first". Prefer this order:

1. Lock C VM experience-memory, MCP gateway, and UTF-8 roundtrips into regression tests.
2. Continue removing or gating Python fallback so failures surface explicitly instead of silently returning to Python semantics.
3. Audit long-running memory ownership and error lifetimes.
4. Keep personal diagnostic scripts out of the repository root unless they are promoted to documented repeatable tools.

**「做完」的验收**（北极星文档原文，逐条可证）：

1. Windows **原生**跑（无 WSL）。
2. 经验层 `experience.ku` / MCP 网关在 C VM 上跑通。
3. 长进程**不漏内存、不 `exit(1)`**。
4. 对拍门禁全绿，且打的是**原生二进制**。

## 1. PM 复盘：上一轮发生了什么

工作树里有一版**未完成的 M5**（`dao/c_vm_bridge.py` + `dao/mcp_server.py` 改动 + 4 个从未绿过的新测试）。

实测诊断（2026-06-15）：

- bridge 直接跑 `1+2` / `思 加一` / `求和` 都**正确**。
- 但 MCP 集成里它把**全部 21 个 std 模块**喂给 C VM → **69 秒延迟**，且**结果被末尾模块尾值盖成 `null`**。
- 第 5 个测试要 sqlite 错误，而 **C VM 根本没有 `sqlite_open`**（M3 未做）→ 只能靠 Python fallback 补。

**PM 判断**：这是 **M5 抢跑到 M3 之前**。它必然又慢又缺能力，唯一能让它「测试绿」的办法是「C VM 失败就回退 Python」——而这恰恰把 **Python 永久焊死成兜底**，与北极星南辕北辙。

> 结论：不修绿这版 M5。把它请下主线（stash 到分支保留），仓库停在 `b5d63f2`，回到真正解锁价值的瓶颈。

## 2. 关键路径（依赖关系）

```
M1（原生构建）──┬──► M3（sqlite + 去 python shell 尾巴）──► M5（翻转，Python 退场）
                └──► M2（结构化错误，已基本完成）
M4（内存所有权）独立大件，可与 M3 并行，M5 前必须收口长进程泄漏
```

**杠杆排序**：M1 > M3 > M5。M4 是必经但可后压。
**M1 是杠杆点**：dev loop 快一个数量级 + 门禁打真实目标 + 物理甩掉 WSL（否则「脱离 Python」只是把依赖从 CPython 换成了 WSL）。

## 3. 里程碑卡片

### M1 — 原生 Windows 构建　`下一步做 · 高杠杆 · 低风险`

- **解决的痛**：当前 C VM 靠 `wsl -d kali-linux` 跑（实测 PATH 里 gcc/cl/clang/clang-cl/tcc **全缺**，`dao/dao_core.exe` **不存在**）。WSL 依赖本身违背「独立站立」。
- **现状**：`dao_core.c` 已加 `_WIN32` 兼容壳、`tools/build_dao_core.ps1` 已就绪（M1 代码层完成），**只差一个能用的本地 C 编译器**。
- **要做**：
  1. 装原生工具链（mingw-w64 `gcc` 或 MSVC Build Tools `cl`）。**这步可能需要你本人执行安装**（见下「需要你定的事」）。
  2. 跑 `tools/build_dao_core.ps1` 产出 `dao/dao_core.exe`。
  3. 让对拍 fixture 实际命中原生 exe（fixture 已写「优先原生、WSL fallback」，需实测确认走原生）。
- **验收**：`dao_core.exe` 在 Windows 直接跑；`pytest tests/test_c_vm_parity.py` 全绿**且打的是原生二进制**（不再 fork WSL）；`verify_core` 不受影响。
- **风险**：低。唯一不确定性是工具链安装是否需要管理员/网络（网络要走 18888 代理）。

### M2 — 结构化错误　`基本完成 · 收尾`

- **现状**：本轮已把 frame 内多处 `exit(1)` 改走 `ExecResult`，仅剩 `main` 顶层 1 处翻译层（符合设计）。
- **要做**：补一遍审计，确认 frame 内无遗漏硬退出；文件读取/import 启动阶段的 stderr 报错是否需要收口，留到 M4 常驻 runtime 时一并处理。
- **验收**：`dao_core.c` 里 `exit()` 仅剩 main 顶层；未捕获错误退出码/形状不变；parity 全绿。

### M3 — sqlite 内置 **+ 去掉经验层的 python shell 尾巴**　`M5 的硬前提 · 中风险`

- **解决的痛**：经验层（`experience.ku`）、任务队列（`task_queue.ku`）是 MCP 网关的核心数据能力，**各 26 处 `sqlite_*` 调用**，C VM 全不支持 → 这部分**永远只能走 Python**。不补 M3，「MCP 走 C VM」就是半截子。
- **隐藏尾巴（PM 必须点名）**：`experience.ku` / `task_queue.ku` 的 `_now()` / `_exp_gen_id()` 用 **`system("python -c ...")`** 取时间戳。即使 sqlite 补齐，C VM 跑经验层时**仍会 fork 出 python.exe**——这是「脱离 Python」的隐藏漏点，必须在 M3 一并消除（C VM 提供原生 `now_str()` / 时间 builtin，改写这两个 `.ku`）。
- **要做**：
  1. `sqlite3.c` amalgamation 进 `vendor/`（零外部依赖，Win/WSL 都能编）。
  2. 实现 `sqlite_open` / `sqlite_exec` / `sqlite_query` / `sqlite_close`，`?` 参数绑定**对齐 Python 侧语义**（见 `runtime.py:1995-2026`：exec 返 rowcount，query 返 dict 列表，open 带连接缓存 + Row factory）。
  3. 补原生时间 builtin，改写 `experience.ku` / `task_queue.ku` 去掉 `system("python …")`。
  4. 用 experience.ku 真实建表/插入/查询路径**对拍 Python DaoVM**。
- **验收**：experience.ku 全部入口在 C VM 与 Python 一致；C VM 跑经验层**不再 fork python.exe**；parity 全绿。

### M4 — 内存所有权（引用计数）　`大件 · 高风险 · M5 前收口`

- **解决的痛**：常驻 runtime 不漏内存（实测 ~71 处分配 / 75 处 free，已有释放但所有权无系统纪律）。
- **已知硬点**：`env ↔ closure ↔ env` 会成环。框架已标出：env→closure 用弱引用断环，或先接受环、后补 mark-sweep。
- **可后压理由**：M1 给了快循环、M2 给了错误传播、ASan 兜底，到这一步工具齐备再动。
- **验收**：典型脚本 + 经验层往返 0 泄漏；parity 全绿。

### M5 — 翻转执行路径，Python 退场　`Phase 5 终点 · 依赖 M3`

- **现状**：上一轮抢跑的版本请下主线后，这里**重做**。
- **正确形态**（与抢跑版的根本区别）：
  - **不是**「C VM 失败就回退 Python」。M3 之后 C VM 已具备 sqlite + 经验层能力，`ku_eval` / `ku_call` **默认且正常走 C VM**。
  - bridge **按需加载** std（靠每个 `.ku` 自带的 `引` import 自动解析依赖，不全量喂 21 个）。
  - Python DaoVM **仅作对拍基准**，不作运行兜底。
- **灰度**：先切 `ku_eval`，对比一致后切 `ku_call`，最后切经验层工具。
- **验收**：MCP 网关默认 C VM；Python DaoVM 仅留对拍基准；4 条北极星全满足。

## 4. 固定工作循环（每个里程碑照做，不变）

```
backup（tar 进 backups/）→ 一个窄改动 → verify_core + parity → 全绿才 after 备份 → 只提交本次相关文件
```

**对拍是地基，永不变红。每个里程碑结束，parity 计数只增不减。**
**不要为了推进度让门禁变红；不要把无关临时文件（wechat_* 等）混进提交。**
